# Chantier D — Internal CRM for inbound + manual leads

**Date:** 2026-05-14
**Status:** Approved design — ready for implementation plan
**Scope:** Persistence layer (Neon Postgres + Drizzle), API for admin CRUD + CSV export, `/admin/leads` Linear-style UI, and a new card on the admin home. Manual lead entry (no email triggered). Updating `/api/lead` to persist before the audit. Out of scope: lead-magnet template editor (chantier C), OnlySystem paper cards editor (chantier A).

## Goal

Stop letting leads vanish after the audit fires. Persist every inbound lead in a Postgres database, let Nik add leads manually, view the audit email sent to each lead, change status, and export to CSV — all from a fast, dense `/admin/leads` page in the existing admin app.

## Non-goals

- Multi-user CRM. Auth stays Nik-only (existing JWT).
- Free-text notes per lead. Skipped — Nik can use another tool today.
- Triggering an audit for manually added leads. Manual = store only.
- Bulk import (CSV upload). Out of scope; the manual form is one-at-a-time.
- Pagination via cursor / virtualized table. Volumes are small; an offset/limit query with a 200-row cap is enough for the foreseeable future.
- Activity log / audit trail per lead. `updated_at` is the only timestamp we track beyond `created_at`.
- Real-time updates (Pusher / WebSockets / Server-Sent Events). The page re-fetches on focus, that's enough.

## Tech stack

- **DB**: Neon Postgres (Free tier, Vercel Marketplace integration). Auto-provisioned env vars: `DATABASE_URL` (pooled, for the app) and `DATABASE_URL_UNPOOLED` (for migrations).
- **Driver**: `@neondatabase/serverless` — HTTP-based driver, no connection pool issues on Vercel functions.
- **ORM**: `drizzle-orm` for typed queries + `drizzle-kit` for migrations.
- **Migrations**: SQL files in `lib/db/migrations/`, generated by `drizzle-kit generate`, applied by `drizzle-kit migrate` (run manually from local with `DATABASE_URL_UNPOOLED`).
- **CSV**: hand-rolled in `lib/csv.ts` — no dependency. Standard `,` separator, `"` quoting, UTF-8 BOM prefix so Excel opens it cleanly.

## Schema

One table, indexes on the three fields we actually query by.

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TABLE leads (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email         TEXT NOT NULL,
  first_name    TEXT,
  phone         TEXT,
  domain        TEXT,
  source        TEXT NOT NULL DEFAULT 'inbound',
  status        TEXT NOT NULL DEFAULT 'new',
  audit_subject TEXT,
  audit_body    TEXT,
  audit_outcome TEXT,
  audit_reason  TEXT,
  signals       JSONB,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX leads_created_at_desc_idx ON leads (created_at DESC);
CREATE INDEX leads_email_idx ON leads (email);
CREATE INDEX leads_status_idx ON leads (status);
```

Constraints in code (not in SQL — keeps migrations cheap):
- `source ∈ {'inbound', 'manual'}`
- `status ∈ {'new', 'contacted', 'client', 'lost'}`
- `audit_outcome ∈ {'success', 'fallback', null}`

Drizzle schema (`lib/db/schema.ts`) mirrors this with `pgTable`, `uuid`, `text`, `timestamp`, `jsonb` helpers, and exports the inferred row types via `typeof leads.$inferSelect` / `$inferInsert`.

## Data flow

### Inbound lead (existing path, modified)

1. `POST /api/lead` validates with zod (unchanged), runs the rate-limit + honeypot + blocked-domain checks (unchanged).
2. **NEW**: `const row = await insertLead({ email, first_name, phone, domain, source: 'inbound' })`. Captures the row's id.
3. `after(() => runAudit({ ...lead, id: row.id }))` — same fire-and-forget pattern. The `Lead` type in `types/audit.ts` gains a new required field `id: string` (a UUID), and `runAudit`'s call sites must thread it through. The audit pipeline's existing internals don't need to read `id`; only the post-pipeline `updateLeadAudit` does.
4. `runAudit` runs the existing pipeline, then:
   - If `outcome === 'success'`: `updateLeadAudit({ id: lead.id, subject, body, outcome: 'success', signals })`.
   - If `outcome === 'fallback'`: `updateLeadAudit({ id: lead.id, subject, body, outcome: 'fallback', reason })`.
5. `sendAuditEmail({...})` runs. If it throws (Resend 4xx/exhausted retries), we log and the lead row stays as-is — `audit_outcome` reflects what the pipeline produced, regardless of whether delivery succeeded.

If the DB insert at step 2 fails (Neon down, bad input, etc.), `/api/lead` returns the same `200 ok` as today — the user shouldn't see the failure. But we log a structured error so it's visible in Vercel runtime logs. The audit pipeline is skipped in that case (no row to update).

### Manual lead

1. `POST /api/admin/leads` with `{ email, first_name?, phone?, domain? }`. JWT-guarded.
2. Validates with zod (`email` required + email-format; others optional strings, trimmed).
3. `insertLead({ ...input, source: 'manual' })`. No audit triggered. Returns the row.
4. Frontend optimistically prepends the new row; on success it replaces the optimistic row with the server's authoritative row.

### Status update

1. `PATCH /api/admin/leads/[id]` with `{ status }` (and optionally `first_name`, `phone`).
2. Validates status against the enum.
3. `updateLeadStatus({ id, status })` (or `updateLeadFields` if other fields included).
4. Returns the updated row.

### Detail

1. Click a row → drawer opens → `GET /api/admin/leads/[id]`.
2. Server returns the full row including `signals` JSONB.
3. Drawer renders all fields. `audit_subject` and `audit_body` shown verbatim in `<pre>` blocks (monospace, preserved whitespace).
   - If `source === 'manual'` and `audit_outcome === null` → the audit section shows a single muted line "Manual entry — no audit was generated."
   - If `source === 'inbound'` and `audit_outcome === null` → "Audit pending — the AI pipeline hasn't finished yet. Refresh in a few seconds."
   - Otherwise (`audit_outcome ∈ {'success','fallback'}`) → render subject + body.
4. `signals` shown collapsed by default (a `<details>` element) with a "View raw" summary. Hidden entirely when null.

### Export

1. Click "Export" → `GET /api/admin/leads/export`.
2. Server fetches all rows (no pagination — Excel-friendly), builds the CSV string via `leadsToCsv`, returns it with `Content-Type: text/csv; charset=utf-8`, `Content-Disposition: attachment; filename="leads-YYYY-MM-DD.csv"`.
3. Columns: `id`, `email`, `first_name`, `phone`, `domain`, `source`, `status`, `audit_outcome`, `created_at` (ISO), `updated_at` (ISO). `audit_subject`, `audit_body`, and `signals` are **excluded** from the CSV — they're verbose, contain newlines that would mangle the CSV, and Nik doesn't need them in a spreadsheet.

## UI design notes (Linear-like)

- **Density**: 36px row height, no row padding above/below text, 14px font.
- **Hover**: subtle row highlight (`#f4f4f5`), pointer cursor, click anywhere on the row.
- **Selection**: clicked row gets a left-border accent + slightly stronger highlight.
- **Status badges**: pill with 1px outline + 10% tint background.
  - `new` → indigo
  - `contacted` → amber
  - `client` → emerald
  - `lost` → zinc-400 (muted)
- **Source pills**: same shape but in zinc-200; just `inbound` or `manual` in 10px caps.
- **Drawer**: 480px wide, slides in from the right with a 200ms ease-out transform, `Esc` to close, click outside the drawer to close. Backdrop is a 20% scrim.
- **Table header**: sticky, semi-transparent backdrop blur — visible scrolling under it.
- **Filters**: status as a segmented control above the table (`All | New | Contacted | Client | Lost`); search by email/domain as a single text input that filters client-side (we never have more than 200 rows on screen, so client-side is fine).
- **Empty state**: dotted-border card with "No leads yet — your next inbound submission will land here."
- **Toolbar buttons**: "+ New lead" (primary) and "Export" (secondary, ghost style).

## Error handling

- **DB unreachable during `/api/lead`**: log with `console.error("[lead] db insert failed", { error })`, return `200 ok` to the client anyway (we never want to leak infra issues to the lead). The audit pipeline is skipped because we have no row to update.
- **DB unreachable during admin requests**: return `500` with a generic message; the UI shows a "Failed to load — retry" state.
- **Stale row at update time**: if `updateLeadAudit` is called for an id that no longer exists (Nik deleted it), log and swallow.
- **CSV with weird characters**: every cell wrapped in `"..."`, quote characters escaped as `""`, newlines stripped from each cell — the export should never break Excel.

## Migration & rollout

1. Add the Neon integration in Vercel dashboard → DB provisioned, env vars set.
2. `vercel env pull` locally.
3. Add Drizzle deps, write schema, generate migration with `drizzle-kit generate`, commit.
4. `drizzle-kit migrate` against `DATABASE_URL_UNPOOLED` locally → schema live on Neon.
5. Build the data-access layer (`lib/db/leads.ts`).
6. Wire `/api/lead` to call `insertLead` before the audit + `updateLeadAudit` from inside `runAudit`.
7. Build the admin API routes.
8. Build the UI.
9. Add the admin home card last (final visible toggle).

Each step ships behind no flag — the table is empty until the first commit lands the inbound write. Until that commit, the admin page just shows "no leads yet".

## Risk register

| Risk | Mitigation |
|---|---|
| Drizzle migration accidentally drops a column on a re-generate | Manual review of each generated SQL file before commit; commit the SQL file, not just the schema TS. |
| Lead inserted but audit pipeline crashes → row stuck with `audit_outcome=null` forever | Acceptable. UI shows "Audit pending" for these rows. A periodic job to re-run is a follow-up, not a v1 requirement. |
| `signals` JSONB column gets huge over time | The Neon Free tier is 500 MB. At 15 KB/row including signals, that's ~30k leads. Far from the cap. |
| Nik deletes a lead by mistake | A confirm dialog on the delete button. No soft-delete in v1 — the table is small enough that a Neon backup recovers anything. |
| `/api/admin/leads/export` includes massive `signals` field, Excel chokes | Already excluded from the CSV. |
| Concurrent writes to the same row (`updateLeadAudit` racing with a Nik manual edit) | `updated_at` defaults to `NOW()` on update. Last write wins. Acceptable for a one-user CRM. |
